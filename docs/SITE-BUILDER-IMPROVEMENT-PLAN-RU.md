# План улучшений AI Site Builder (детальный)

Документ фиксирует **направление развития** продукта на основе текущей кодовой базы (`server.js`, оркестратор `src/agent/`, сборщик `src/builder/`, медиамодуль, UI). Цель — **выше качество сайтов**, **ниже себестоимость токенов**, **лучший UX**, **работающие формы и динамика** там, где это нужно пользователю.

**Версия:** 2026-05-06 — согласовано с уже реализованным overhaul (Brief → Architect → Context7 → Coder с tools → Smoke Playwright → Autofix → Reviewer).

## 2. Принципы приоритизации

1. **Максимальный эффект на юнит токена** — шаблоны и ограниченный дизайн-словарь сильнее любого усложнения стека без них.
2. **Безопасность и предсказуемость** — любой код, который выполняется на сервере (сборки, sandbox), должен быть изолирован по процессам/директориям (у вас уже есть `projects/<id>/` и кэш `node_modules`).
3. **Совместимость с текущим оркестратором** — новые возможности встраиваются как **стадии, tools или ветки Architect**, а не как отдельный «второй генератор».
4. **Измеримость** — каждая фаза плана имеет метрики успеха (см. раздел 9).

## 3. Стратегические направления (пиллары)

### P1 — Дизайн-система и шаблоны (главный рычаг качества)

**Проблема:** модель каждый раз изобретает типографику, отступы, карточки и hero с нуля → визуальная нестабильность и перерасход токенов.

**Решение:**

- Каталог **5–7 стартовых шаблонов** (лендинг услуг, SaaS, портфолио, локальный бизнес, простая игра, личный блог, мини-e-commerce без оплаты).
- Внутри шаблонов — **переиспользуемые блоки**: hero, pricing, галерея, отзывы, FAQ, контакты, форма лидогенерации, footer.
- Библиотека уровня **shadcn/ui + Tailwind tokens** (цветовые переменные, радиусы, типошкала) — либо как **реальный** набор компонентов в `react-bundle`, либо как **документированный HTML-фрагменты** для static-режима с жёсткими правилами в system prompt.

**Интеграция в продукт:**

- **Architect** (`src/agent/stages/architect.js`): поле `templateId` + `variant` (тёмная/светлая, акцентный цвет из брифа).
- **Coder**: в fresh-mode сначала `write_file` из заготовок репозитория (`src/builder/presets/<templateId>/`), затем правки через `apply_patch`.
- **Reviewer** (`src/agent/stages/reviewer.js`): чеклист сверки с выбранным шаблоном (не «абстрактный премиум», а «совпадение с эталоном секций»).

**Оценка трудозатрат:** 3–5 рабочих дней на первый набор из 5 шаблонов + правки промптов.

**Риски:** раздувание репозитория — держать пресеты компактными, не копировать целые node_modules в git.

---

### P2 — Vision-вход (референс → структура и стиль)

**Проблема:** пользователю проще показать скрин, чем описать «как на Apple».

**Решение:**

- Новый этап **Vision Brief** (опционально): если в запросе есть изображение (загрузка в чат), вызывается vision-capable модель (Claude / GPT-4o) с задачей: извлечь **палитру, иерархию секций, тип кнопок, плотность контента** — структурированный JSON для Architect.
- Не копировать пиксель-в-пиксель (юридически рискованно), а **переносить паттерны**.

**Интеграция:**

- UI: вложение файла в сообщение → `multipart` на `/api/generate/stream` или отдельный `/api/vision-brief`.
- Оркестратор: после Brief — merge `visionHints` в plan.

**Оценка:** 1–2 дня (API + хранение вложений + один vision-вызов).

**Риски:** стоимость vision-токенов — ограничить размер изображения (resize на сервере до 1024px).

---

### P3 — Динамика: формы, каталоги, «мини-бэкенд»

**Проблема:** статический `index.html` не сохраняет заявки и не ведёт каталог в БД.

**Варианты (от дешёвого к мощному):**

| Вариант | Плюсы | Минусы | Когда |
|---------|--------|--------|------|
| **A. Formspree / Getform / Basin** (внешний endpoint) | Почти нулевая разработка | Зависимость от третьей стороны, лимиты | MVP форм |
| **B. PocketBase** на том же VPS | SQLite, auth, файлы, realtime, один бинарь | Операционные задачи (бэкапы, обновления) | Универсальный «наш бэкенд» |
| **C. Supabase** (облако) | Postgres, Row Level Security, UI | Стоимость, привязка аккаунта пользователя | Когда нужен «настоящий» SaaS |
| **D. Cloudflare Workers + D1/KV** | Edge, дёшево на трафике | Другой стек деплоя | JAMstack на CF |

**Рекомендуемая траектория для вашего проекта:**

1. Фаза 3.1: **шаблон «форма → webhook»** (настраиваемый URL или встроенный прокси в `server.js`, пишущий в SQLite/файл по `projectId`).
2. Фаза 3.2: **PocketBase** как опциональный профиль «Сайт с личным кабинетом» — Architect выставляет `backend: pocketbase`, Coder генерит клиент SDK + env-плейсхолдеры.
3. Фаза 3.3: экспорт схемы для **Supabase** (SQL миграция + инструкция) для продвинутых пользователей.

**Оценка:** 3–6 дней на 3.1–3.2 в минимальном виде.

**Риски:** безопасность — никогда не класть секреты пользователя в сгенерированный публичный репозиторий; разделять server-side keys и public anon keys.

### P4 — Превью и сборка: WebContainers vs текущий esbuild

**Текущее:** `esbuild-runner` + опционально static CDN — достаточно для большинства лендингов.

**Улучшение UX:**

- **StackBlitz WebContainers** (в браузере пользователя): мгновенный `npm run dev` без нагрузки на ваш VPS. Подходит для демо «настоящего» React/Vite проекта.
- **Серверная песочница (E2B, Daytona, Firecracker)** — если агенту нужен полный `npm test`, e2e, произвольные команды. Дороже и сложнее в эксплуатации.

**Рекомендация:** сначала **улучшить текущий bundle-поток** (кэш, логи сборки в UI, кнопка «скачать zip»). WebContainers — как **отдельная вкладка превью** для power users. E2B — только если появятся запросы на произвольный Node в агенте.

**Оценка WebContainers:** 2–4 дня интеграции UI + политика CSP.

---

### P5 — Мульти-кандидаты (A/B генерация)

**Идея:** параллельно 2 варианта (например, OpenAI + OpenRouter Claude), пользователь выбирает карточку в UI, второй отбрасывается или сохраняется как «черновик».

**Плюсы:** выше вероятность «попасть» в ожидание пользователя. **Минусы:** ~2× стоимость на этот шаг.

**Интеграция:** флаг в запросе `variants: 2`, оркестратор запускает два `runCoder` в **разные подкаталоги** `projects/<id>/variant-a|b`, smoke на обоих, UI — сравнение скриншотов Playwright.

**Оценка:** 2–3 дня.

---

### P6 — Экономия токенов

| Механизм | Описание | Где в коде |
|----------|-----------|------------|
| Кэш типовых блоков | Embeddings (text-embedding-3-small / локальный) для поиска похожего hero/footer из библиотеки | Новый модуль + tool `suggest_block` |
| Дешёвые модели для Brief/Reviewer | Уже частично; расширить список «лёгких» моделей | `stages/brief.js`, `reviewer.js` |
| Ужесточение patch-mode | Более точная эвристика + принудительный patch для follow-up < N токенов | `orchestrator.js` |
| Сжатие контекста | Для больших проектов — передавать в Coder только `tree + выжимки файлов`, не полные файлы | `coder.js`, tool `read_file` |

**Оценка:** итерациями по 0.5–1 день.

---

### P7 — Редактирование кода пользователем (без code-server)

**Почему не [code-server](https://github.com/coder/code-server):** это полноценный VS Code в браузере для людей; у агента нет стабильного API для «управления IDE». Для вашего сценария избыточно по RAM и по модели угроз (долгоживущие процессы, расширения).

**Альтернатива:** встроить **Monaco Editor** в `public/` для `projects/<id>/` файлов — чтение/запись через уже существующие API или новый `PATCH /api/projects/:id/file`.

**Оценка:** 1–2 дня для read-only Monaco; 3–5 дней с безопасным save и валидацией.

## 5. Изменения по подсистемам (чеклист для разработки)

### 5.1 Backend / `server.js`

- Новые маршруты: `POST /api/projects/:id/leads` (или общий webhook), `GET /api/projects/:id/export.zip`.
- Лимиты размера тела, rate limit по IP/сессии для публичных форм.
- При появлении PocketBase — reverse proxy или отдельный поддомен `pb.*` с TLS.

### 5.2 Оркестратор

- Расширить `meta` событие: `templateId`, `backendProfile`, `visionAttached`.
- При multi-variant — подкаталоги и агрегация smoke-результатов.

### 5.3 Tools

- `list_presets`, `init_from_preset` (копирование файлов из git-tracked пресета).
- `submit_lead_schema` — опционально, для typed forms.

### 5.4 UI

- Выбор шаблона до отправки промпта (или авто + «сменить шаблон»).
- Загрузка изображения референса.
- Карточка сравнения A/B (если включено).

### 5.5 Наблюдаемость

- Логировать: `templateId`, `modelUsed`, `tokens`, `smokeOk`, `durationMs`, `transportRetries`.
- Дашборд (хотя бы CSV/агрегация в существующей статистике токенов).

## 6. Риски и митигация

| Риск | Митигация |
|------|-----------|
| Юридический (копирование чужих сайтов с vision) | Явный disclaimer + стилистический перенос, не клон |
| Утечка API-ключей в сгенерированном фронте | Разделение `PUBLIC_*` и server-only env; сканер в smoke |
| Перегруз VPS от сборок | Очередь, лимит параллельных `esbuild`, TTL кэша (уже частично) |
| Сложность поддержки пресетов | Версионирование пресетов `v1`, `v2` |

## 7. Что сознательно откладываем

- Полная замена генератора на **code-server** — несоответствие задаче (см. P7).
- Произвольный shell в агенте без sandbox — безопасность.
- Обучение собственной модели — вне текущего scope.

## 9. Метрики успеха (рекомендуемые KPI)

| Метрика | Как считать | Цель (ориентир) |
|---------|-------------|-----------------|
| **Smoke pass rate** | % проектов с `smoke.ok` после первого прогона | > 85% |
| **Средний output tokens на проект** | из `_stats.json` / БД | −30% после пресетов |
| **NPS по «визуалу»** | опрос в UI 1–5 | рост на +0.5 за квартал |
| **Время до первого превью** | p95 от запроса до `done` | < 90s для static, < 3min для bundle |
| **Повторные генерации из-за ошибки** | доля сессий с >1 полным прогоном | < 20% |

## 10. Резюме для стейкхолдеров

1. **Самый сильный рычаг** — пресеты + дизайн-система (P1), не новый runtime.  
2. **Vision** (P2) — быстрый wow-эффект и меньше «размытых» промптов.  
3. **Формы и БД** (P3) — отделяет «игрушечный лендинг» от рабочего инструмента для бизнеса.  
4. **WebContainers / E2B** (P4) — по мере роста требований к full-stack, не обязательны на старте Q1.  
5. **A/B и embeddings** (P5–P6) — оптимизация конверсии и себестоимости после стабилизации базового качества.
